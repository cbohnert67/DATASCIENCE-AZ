Chapitre 31 : Git & GitHub pour les Data Scientists - Collaborez sans tout casser

Imaginez que vous travaillez sur votre projet de prédiction. Pour sauvegarder vos progrès, vous créez des copies de votre notebook : `projet_v1.ipynb`, `projet_v2_nettoyage.ipynb`, `projet_final.ipynb`, `projet_final_VRAIMENT_final.ipynb`... C'est le chaos assuré. Et que se passe-t-il si vous voulez collaborer avec un ami ? Vous vous envoyez les fichiers par email ? Git est le système de contrôle de version professionnel conçu pour éviter ce désordre, suivre l'historique de vos modifications et collaborer efficacement.

L'art de voyager dans le temps (et dans le code)

Pourquoi le versioning est-il crucial en Data Science ?

En Data Science, un projet n'est pas seulement du code, c'est une succession d'expériences. Le contrôle de version est la seule façon de tenir un journal de bord rigoureux de ces expériences.

Garantir la reproductibilité

Le scénario catastrophe : Six mois après avoir fini un projet, un client vous demande : "Pouvez-vous régénérer le modèle exact qui nous a donné 92.4% de précision ?". Sans Git, c'est presque impossible. Vous avez changé les versions des bibliothèques, légèrement modifié une fonction de nettoyage... Avec Git, il vous suffit de retrouver le `commit` correspondant à ce résultat pour restaurer l'intégralité de l'environnement et du code. C'est la base de la rigueur scientifique.

Revenir en arrière en toute sécurité

Git est le "Ctrl+Z" ultime pour tout votre projet. Vous avez testé une nouvelle technique de feature engineering qui s'avère catastrophique et a cassé la moitié de votre code ? Pas de panique. Une simple commande (`git checkout` ou `git revert`) vous permet de revenir à la dernière version stable de votre projet, comme si de rien n'était.

Expérimenter sans risque avec les branches

C'est l'une des fonctionnalités les plus puissantes. Imaginez que votre branche `main` est votre projet stable et fonctionnel. Vous voulez tester un nouvel algorithme comme XGBoost ?

git checkout -b experiment-xgboost

Cette commande crée une "copie" de votre projet, une branche nommée `experiment-xgboost`, où vous pouvez faire toutes les modifications que vous voulez. La branche `main` reste intacte. Si votre expérience est un succès, vous pourrez fusionner (`merge`) vos changements dans `main`. Si c'est un échec, vous supprimez simplement la branche. Fini les `projet_v2_test_xgboost.ipynb` !

La différence entre Git et GitHub

L'analogie la plus simple est celle-ci : Git est le logiciel Word, GitHub est Google Docs.

Git : L'atelier local

Git est un programme qui s'exécute sur votre ordinateur. Il ne sait rien du web. Son seul travail est de suivre les modifications de vos fichiers dans un dossier (votre dépôt) et de vous permettre de créer des commits, de créer des branches, etc. Toutes les commandes comme `git add`, `git commit`, `git status` se passent à 100% sur votre machine.

GitHub : La plateforme de partage

GitHub est un site web qui héberge des dépôts Git. C'est une sorte de "cloud" pour votre code. Son but est de faciliter la collaboration et la sauvegarde. C'est sur GitHub que vous pouvez :

  • Stocker une copie de sauvegarde de votre dépôt.
  • Partager votre code avec le monde entier (ou en privé).
  • Collaborer avec d'autres via les "Pull Requests".
  • Gérer votre projet (suivi de bugs, etc.).

Les commandes `git push`, `git pull`, `git clone` sont les ponts qui connectent votre Git local à votre dépôt distant sur GitHub.

Le workflow du Data Scientist avec Git

Les concepts de base

  • Dépôt (Repository) : C'est simplement le dossier de votre projet. Quand vous lancez `git init` dedans, Git commence à le surveiller.
  • Commit : C'est un "point de sauvegarde" intelligent. Plutôt que de copier tout le projet, il enregistre uniquement les différences par rapport au commit précédent. Chaque commit a un identifiant unique et un message descriptif, formant une chaîne d'historique. Un bon message de commit est crucial : `git commit -m "Ajout du modèle RandomForest"` est bien meilleur que `git commit -m "modif"`.
  • Branche (Branch) : C'est une copie de travail de votre projet. La branche principale s'appelle `main`. Quand vous créez une nouvelle branche, vous créez un espace de travail parallèle où vous pouvez expérimenter sans affecter la version stable. C'est comme faire une photocopie d'un document pour y prendre des notes, tout en gardant l'original intact.

Le cycle de vie d'une modification (Les 3 étapes)

Le workflow pour enregistrer une modification est un processus en trois temps, comme préparer un colis :

[Image d'un diagramme du workflow Git : 1. Working Directory -> 2. Staging Area -> 3. Local Repository] 
# Vous travaillez sur vos fichiers (`analyse.ipynb`, `utils.py`...)
# C'est votre "Bureau de travail" (Working Directory).

# ÉTAPE 1 : Sélectionner ce qui va dans le colis
$ git add analyse.ipynb
# Vous mettez `analyse.ipynb` dans le "carton" (Staging Area).
# Le fichier `utils.py` sur lequel vous travaillez encore reste sur le bureau.

# ÉTAPE 2 : Sceller et étiqueter le colis
$ git commit -m "Ajout de la visualisation des corrélations"
# Vous fermez le carton et vous mettez une étiquette claire.
# Le "commit" est maintenant sauvegardé dans votre dépôt local (Local Repository).

Le workflow complet avec GitHub (Collaboration)

Voici le cycle complet quand vous travaillez en équipe :

# 1. Obtenir le projet (une seule fois au début)
$ git clone https://github.com/equipe/projet-titanic.git

# 2. Avant de commencer à travailler, TOUJOURS se mettre à jour
$ git pull
# Récupère les modifications faites par vos collègues pendant la nuit.

# 3. Vous travaillez sur votre notebook...

# 4. Vérifier où vous en êtes (votre meilleur ami)
$ git status
On branch main
Changes not staged for commit:
  (use "git add ..." to update what will be committed)
  modified:   analyse.ipynb

# 5. Sauvegarder votre travail localement (add + commit)
$ git add analyse.ipynb
$ git commit -m "Correction du bug dans le calcul de l'âge moyen"

# 6. Partager votre travail avec l'équipe
$ git push
# Envoie votre nouveau commit sur le dépôt distant sur GitHub.

L'arme secrète du Data Scientist : .gitignore

Imaginez que vous lancez la commande `git status` et que vous voyez une liste interminable de fichiers : vos datasets, des fichiers temporaires, des dossiers de configuration... C'est le chaos. Le fichier `.gitignore` est votre agent de nettoyage : il indique à Git "Ne t'occupe pas de ces fichiers, ignore-les". C'est un simple fichier texte nommé `.gitignore` (le point au début est crucial) que vous placez à la racine de votre projet.

Avant / Après : L'impact d'un `.gitignore`

Sans `.gitignore`

$ git status
Untracked files:
  (use "git add ..." to include in what will be committed)
    .DS_Store
    .env
    data/titanic.csv
    venv/
    analyse.ipynb
    .ipynb_checkpoints/

Votre statut est pollué par des fichiers inutiles ou sensibles.

Avec `.gitignore`

$ git status
Untracked files:
  (use "git add ..." to include in what will be committed)
    analyse.ipynb

C'est clair ! Vous ne voyez que les fichiers pertinents de votre projet.

Exemple de `.gitignore` commenté pour un projet de Data Science

# Fichiers de données. Ne JAMAIS versionner les données brutes.
# Partagez-les via un stockage cloud et décrivez-les dans le README.
*.csv
*.json
*.parquet
data/

# Fichiers de configuration et secrets. Contient les clés d'API, mots de passe...
# C'est une faille de sécurité majeure de les versionner.
.env
*.env

# Dossiers d'environnement virtuel Python.
# Ils sont spécifiques à votre machine et peuvent peser des centaines de Mo.
venv/
env/

# Fichiers temporaires des notebooks Jupyter.
.ipynb_checkpoints/

# Fichiers cache de Python.
__pycache__/

# Fichiers système spécifiques à l'OS (macOS, Windows).
.DS_Store
Thumbs.db
